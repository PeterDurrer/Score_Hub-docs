Berechtigungskonzept (Permission System) ​

Übersicht ​

Das Berechtigungssystem von Score Hub implementiert ein rollenbasiertes Zugriffskontrollsystem (Role-Based Access Control, RBAC) mit drei Ebenen:

  1. Benutzerrollen (User Roles) - Steuern den globalen Menüzugriff
  2. Vereinsrollen (Club Roles) - Steuern vereinsspezifische Operationen
  3. Wettkampfrollen (Competition Roles) - Steuern wettkampfspezifische Operationen

Permission Context ​

typescript
interface PermissionContext {
  user: AuthenticatedUser;
  clubID?: string;           // Optional: für vereinsspezifische Berechtigungen
  competitionID?: string;    // Optional: für wettkampfspezifische Berechtigungen
}

Backend Beispiele ​

Beispiel 1: API Route mit Berechtigungsprüfung ​

typescript
import { canInsertIntoTable, canModifyRecord } from "../utils/permissions";

// Prüfen, ob Benutzer Schiessstand hinzufügen kann
async function createClubLocation(
  request: HttpRequest,
  context: InvocationContext,
  user: AuthenticatedUser
): Promise<HttpResponseInit> {
  const body = await request.json();
  const clubID = body.clubID;
  
  // Prüfe Berechtigung
  const hasPermission = await canInsertIntoTable("clubLocations", {
    user,
    clubID,
  });
  
  if (!hasPermission) {
    return forbiddenResponse("Keine Berechtigung zum Hinzufügen von Schiessständen");
  }
  
  // Führe Operation aus...
}

Frontend (React) ​

Das Frontend nutzt Hooks und Helper-Funktionen für Berechtigungsprüfungen:

usePermission Hook ​

typescript
// In einer Komponente
import { usePermission } from "../hooks/usePermission";

function ClubLocationsList({ clubID }: { clubID: string }) {
  const { canInsert, canModify } = usePermission();
  
  const canAddLocation = canInsert("clubLocations", { clubID });
  const canEditLocation = (locationID: string) => 
    canModify("clubLocations", locationID, { clubID });
  
  return (
    <div>
      {canAddLocation && (
        <button onClick={handleAddLocation}>
          Neuer Schiessstand
        </button>
      )}
      
      {locations.map(location => (
        <div key={location.id}>
          {location.name}
          {canEditLocation(location.id) && (
            <button onClick={() => handleEdit(location.id)}>
              Bearbeiten
            </button>
          )}
        </div>
      ))}
    </div>
  );
}

Best Practices ​

Backend ​

  1. Immer Berechtigungen prüfen: Jede API-Route sollte Berechtigungen prüfen, bevor sie Operationen ausführt.

  2. Context mitgeben: Stelle sicher, dass der richtige Context (clubID, competitionID) mitgegeben wird.

  3. Früh prüfen: Prüfe Berechtigungen am Anfang der Funktion, bevor Du Datenbank-Operationen ausführst.

typescript
// Gut ✓
async function updateRecord(request, context, user) {
  const id = request.params.id;
  
  // Früh prüfen
  if (!await canModifyRecord("table", id, { user })) {
    return forbiddenResponse();
  }
  
  // Dann Operation ausführen
  await updateInDatabase(id);
}

// Schlecht ✗
async function updateRecord(request, context, user) {
  const id = request.params.id;
  
  // Erst Operation, dann prüfen
  await updateInDatabase(id);
  
  if (!await canModifyRecord("table", id, { user })) {
    // Zu spät!
    return forbiddenResponse();
  }
}

Frontend ​

  1. UI-Elemente ausblenden: Verstecke Buttons und Menüpunkte, wenn der Benutzer keine Berechtigung hat.

  2. Benutzerfreundliche Meldungen: Zeige klare Fehlermeldungen, wenn eine Berechtigung fehlt.

  3. Optimistische UI: Prüfe Berechtigungen im Frontend, aber verlasse Dich auf Backend-Validierung.

Erweiterung des Systems ​

Neue Tabelle hinzufügen ​

  1. Füge die Tabelle zu TABLE_PERMISSIONS in api/src/utils/permissions.ts hinzu:
typescript
const TABLE_PERMISSIONS = {
  // ...
  newTable: {
    read: ["admin", "user"],
    insert: ["admin"],
    modify: ["admin"],
  },
};
  1. Dokumentiere die Berechtigungen in dieser Datei.

Neue Rolle hinzufügen ​

  1. Füge die Rolle zum entsprechenden Enum hinzu:
typescript
export enum ClubRole {
  ADMIN = "admin",
  COMPETITION_ADMIN = "competitionAdmin",
  RESULT_SCORER = "resultScorer",
  NEW_ROLE = "newRole", // Neue Rolle
}

  1. Aktualisiere TABLE_PERMISSIONS mit der neuen Rolle.

  2. Dokumentiere die neue Rolle und ihre Berechtigungen.

Fehlerbehandlung ​

Backend ​

typescript
// Unauthorized (401) - Benutzer ist nicht angemeldet
return unauthorizedResponse("Sie müssen angemeldet sein");

// Forbidden (403) - Benutzer hat keine Berechtigung
return forbiddenResponse("Keine Berechtigung für diese Operation");

Frontend ​

typescript
// Fehler abfangen und anzeigen
try {
  await api.updateCompetition(id, data);

    toast.error("Keine Berechtigung zum Bearbeiten dieses Wettkampfs");
  }
}

FAQ ​

Wer kann Wettkämpfe erstellen? ​

  • Global admins (user.role = "admin")
  • Club admins (clubUser.role = "admin")
  • Competition admins (clubUser.role = "competitionAdmin")

Wer kann Resultate erfassen? ​

  • Global admins
  • Club: admin, competitionAdmin, resultScorer
  • Competition: admin, resultScorer

Wer kann Ranglisten sehen? ​

Alle Benutzer mit Zugriff auf den Wettkampf (inkl. viewer).

Wie wird der Wettkampf-Ersteller als Admin eingetragen? ​

Beim Erstellen eines Wettkampfs wird automatisch ein Eintrag in competitionUsers mit role = "admin" für den Ersteller angelegt.

Migration ​

Beim Upgrade auf das neue Berechtigungssystem:

  1. Bestehende Daten: Stelle sicher, dass alle clubUsers und competitionUsers korrekte Rollenwerte haben.

  2. Frontend-Updates: Aktualisiere alle Komponenten, um die neuen Permission-Hooks zu verwenden.

  3. Backend-Updates: Füge Berechtigungsprüfungen zu allen API-Routes hinzu.

  4. Tests: Teste alle Szenarien mit verschiedenen Rollen.
